# Анализ инцидента: Ошибки пайплайна перевода выделенного текста

В этом документе собран полный технический разбор проблем, с которыми столкнулся разработчик при использовании функции перевода выделенного текста (каскадный пайплайн), а также перечень уже внесённых правок и нюансов, требующих дальнейшего исправления.

---

## 1. Исходная проблема (Со слов разработчика)

Ситуация, описанная во время тестирования:

> "Проанализируй проблему, пожалуйста. У меня Google Translate не работает. Я нажимал, нажал Space. У меня открылась командная строка cmd, новая окошко. И с путем в хэнди. `D:\dev\Handy>` И включилась запись голоса, хотя она не должна включаться, потому что вообще ползунка не нашел, как отключить этот голос, Которые включаются через контрол Шифт Спэйс. Я зашел в программу, после этого, а у меня как-будто ползунок translate output сам выключился.. не понимаю.. Пожалуйста, проверь все хорошенько, почему так работает криво."

---

## 2. Глубокий анализ: Почему система повела себя именно так?

Поведение программы показалось аномальным, но при детальном разборе логики оказалось, что оно является следствием наложения сразу **трёх независимых факторов**, каждый из которых усилил эффект другого.

### А) Почему ползунок "Translate output" выключался сам по себе?

**Корневая причина:** Интерфейс использовал оптимистичное обновление (optimistic UI) — при переключении ползунка TypeScript-состояние в Zustand-сторе обновлялось мгновенно, и интерфейс показывал, что перевод включён.

Однако **на стороне Rust-бэкенда не были реализованы Tauri-команды** для сохранения настроек перевода:
- `change_translation_enabled_setting`
- `change_translation_service_setting`
- `change_translation_target_language_setting`

Из-за этого настройка не записывалась в файл хранилища (`settings_store.json`). При **первом же перезапросе настроек с бэкенда** (который происходит при перерендеринге компонента, при событии `model-state-changed`, или при скрытии/показе окна) — фронтенд получал объект `AppSettings` из Rust, в котором `translation_enabled` имел **дефолтное значение `false`** (определено в [`settings.rs:626-628`](file:///d:/dev/Handy/src-tauri/src/settings.rs#L626-L628)). Zustand послушно переписывал UI-состояние, выключая ползунок обратно.

**Файлы:** [`settings.rs:410-415`](file:///d:/dev/Handy/src-tauri/src/settings.rs#L410-L415), [`settingsStore.ts:144-149`](file:///d:/dev/Handy/src/stores/settingsStore.ts#L144-L149)

### Б) Почему открылась командная строка (`D:\dev\Handy>`)

**Корневая причина:** Тестирование проводилось из активного окна терминала (в котором был запущен процесс `bun run tauri dev`).

Для получения выделенного текста программа вызывает `try_get_selected_text()` в файле [`clipboard.rs:15-44`](file:///d:/dev/Handy/src-tauri/src/clipboard.rs#L15-L44), который симулирует нажатие клавиш `Ctrl+C` через обёртку `input::send_copy_ctrl_c()` (модуль [`input.rs`](file:///d:/dev/Handy/src-tauri/src/input.rs)).

В большинстве приложений (Блокнот, Word, Браузер) `Ctrl+C` копирует выделенный текст. Но в терминалах (cmd.exe, PowerShell) `Ctrl+C` интерпретируется как **команда прерывания процесса** (SIGINT). Следовательно, симуляция `Ctrl+C` внутри терминала:
1. Прервала текущую команду (выведя новое приглашение `D:\dev\Handy>`)
2. **Не скопировала** никакой текст в буфер обмена

> **Замечание:** Документ ранее указывал, что симуляция идёт напрямую через библиотеку `enigo` из файла `clipboard.rs`. На самом деле `clipboard.rs` делегирует вызов в `input.rs` → `send_copy_ctrl_c()`, который уже использует `enigo`. Это важно для понимания архитектуры при отладке.

### В) Почему включилась запись голоса, хотя ожидался перевод текста?

Это следствие **совместного действия** проблем А и Б. Логика в [`TranscribeAction::start()`](file:///d:/dev/Handy/src-tauri/src/actions.rs#L522-L546) (файл `actions.rs:526-546`) работает так:

```
1. Если self.post_process == true:
2.   has_pipeline = post_process_enabled || (translation_enabled && !target_language.is_empty())
3.   Если has_pipeline == true:
4.     Попытаться получить выделенный текст (try_get_selected_text)
5.     Если текст получен → запустить каскадный пайплайн
6.     Если текст НЕ получен → перейти к записи голоса
7.   Если has_pipeline == false:
8.     Сразу перейти к записи голоса (без попытки получить текст!)
```

В данном случае:
- Из-за бага А: `translation_enabled == false` (настройка не сохранилась)
- `post_process_enabled` тоже `false` (LLM постобработка явно не включалась)
- ⇒ `has_pipeline == false`, поэтому код **сразу перешёл на шаг 7** — запись голоса, даже не попытавшись проверить clipboard

Даже если бы `has_pipeline` был `true`, из-за бага Б буфер обмена был пуст (Ctrl+C в терминале = SIGINT), и `try_get_selected_text()` вернул бы `None`, что всё равно привело бы к записи голоса (шаг 6).

**Итого:** Баги А и Б дублировали друг друга, гарантируя одинаковый результат двумя разными путями.

## 3. Что уже было исправлено (Внесённые правки в код)

Экстренно внесены исправления для синхронизации настроек перевода между фронтендом и бэкендом:

1. **Бэкенд (Rust)**
   - В [`shortcut/mod.rs:1063-1096`](file:///d:/dev/Handy/src-tauri/src/shortcut/mod.rs#L1063-L1096) добавлены три Tauri-команды:
     - `change_translation_enabled_setting` — сохраняет флаг включения/выключения перевода
     - `change_translation_service_setting` — сохраняет выбранный сервис перевода (`google` / `llm`)
     - `change_translation_target_language_setting` — сохраняет целевой язык
   - Все три команды зарегистрированы в макросе `collect_commands!` файла [`lib.rs:365-367`](file:///d:/dev/Handy/src-tauri/src/lib.rs#L365-L367).

2. **Фронтенд (TypeScript)**
   - В файле [`settingsStore.ts:144-149`](file:///d:/dev/Handy/src/stores/settingsStore.ts#L144-L149) в объект `settingUpdaters` добавлены три записи, привязывающие ключи `translation_enabled`, `translation_service`, `translation_target_language` к соответствующим Tauri-командам.
   - В `src/bindings.ts` (автогенерируемый файл Specta) вручную добавлены TypeScript-типы для новых команд (чтобы TypeScript не выдавал ошибки без полной пересборки).

## 4. Найденные неучтённые нюансы и ошибки ("Подводные камни")

В результате глубокого аудита кода обнаружена серия дополнительных проблем. Статус каждой отмечен индивидуально.

### 4.1 ✅ Бессмысленная вставка того же самого текста — ИСПРАВЛЕНО

- **Файл:** [`actions.rs:528-537`](file:///d:/dev/Handy/src-tauri/src/actions.rs#L528-L537), функция `start_process_selected_text`
- **Суть:** Если и LLM-обработка, и перевод возвращают ошибку (или оба отключены), переменная `result` остаётся равной исходному `selected_text`. Раньше код проверял только `!result.is_empty()` и вставлял текст обратно.
- **Решение:** Добавлена проверка `if result == selected_text` — при совпадении paste пропускается, UI сбрасывается. При ошибке в пайплайне воспроизводится звуковой сигнал.

### 4.2 ✅ Лимит длины URL в Google Translate (HTTP 414) — ИСПРАВЛЕНО

- **Файл:** [`google_translate.rs:16`](file:///d:/dev/Handy/src-tauri/src/google_translate.rs#L16)
- **Суть:** Реализация использовала GET-запрос — весь переводимый текст передавался в строке URL как query-параметр `q`. Google отклонял URL длиннее ~8192 символов.
- **Решение:** Заменён `client.get(url).query(&params)` на `client.post(url).form(&params)`. POST с `application/x-www-form-urlencoded` не имеет ограничения длины URL.

### 4.3 ⏭️ Перевод игнорируется при обычном шорткате (Ctrl+Space) — НЕ ТРЕБУЕТСЯ

- **Суть:** Перевод работает только через `Ctrl+Shift+Space`, а не через `Ctrl+Space`.
- **Решение:** Оставлено как есть — это осознанное архитектурное разделение. `Ctrl+Space` = быстрая транскрипция без задержек. `Ctrl+Shift+Space` = полный каскад (постобработка + перевод). UI уже указывает привязку к шорткату. Пользователь знает, что перевод выделенного текста — через `Ctrl+Shift+Space`.

### 4.4 ✅ Тихие ошибки при сбоях сети/API — ЧАСТИЧНО ИСПРАВЛЕНО

- **Файлы:** [`actions.rs:275-361`](file:///d:/dev/Handy/src-tauri/src/actions.rs#L275-L361) (функция `translate_transcription`), [`actions.rs:528-537`](file:///d:/dev/Handy/src-tauri/src/actions.rs#L528-L537) (функция `start_process_selected_text`)
- **Суть:** При ошибке Google Translate / LLM пайплайн возвращал исходный текст без уведомления.
- **Решение:** При ошибке (результат не изменился + had_error == true) воспроизводится звуковой сигнал через `play_feedback_sound`. Toast/notification пока не реализован.

### 4.5 ⏭️ Захардкоженный промпт для LLM-перевода — НЕ ТРЕБУЕТСЯ

- **Суть:** Промпт для `TranslationService::Llm` вшит в Rust-код и не редактируется из UI.
- **Решение:** Не актуально — используется только Google Translate, LLM-перевод не планируется. Если в будущем понадобится LLM, можно будет добавить настраиваемое поле `translation_prompt` в `AppSettings`.

### 4.6 ✅ `start_process_selected_text` не привязан к истории — ИСПРАВЛЕНО

- **Файл:** [`actions.rs:514-526`](file:///d:/dev/Handy/src-tauri/src/actions.rs#L514-L526)
- **Суть:** Функция обрабатывала выделенный текст, но не сохраняла результат в БД.
- **Решение:** Добавлен вызов `hm.save_entry(...)` с пустым `file_name` (нет WAV), исходным текстом как `transcription_text`, и результатами LLM/перевода. Запись появляется в истории наравне с голосовыми транскрипциями.

### 4.7 ✅ Гонка состояний re-paste после Ctrl+C — ИСПРАВЛЕНО

- **Файл:** [`clipboard.rs:15-57`](file:///d:/dev/Handy/src-tauri/src/clipboard.rs#L15-L57)
- **Суть:** Фиксированная задержка 150ms была недостаточна для медленных приложений (Electron, IDE).
- **Решение:** Реализован polling: clipboard проверяется каждые 50ms, максимум до 500ms. Быстрые приложения отвечают мгновенно (один цикл), медленные получают до 500ms. При раннем обнаружении текста цикл прерывается — нет лишней задержки.
